# Validation Contract

## 1. Purpose

The validation contract defines how TSB artifacts are checked for structural validity, behavioral compatibility, and leaderboard eligibility.

It applies to:

- scenarios
- world-state artifacts
- tool manifests
- traces
- score reports
- submissions

## 2. Validator classes

TSB SHOULD provide the following validators:

- schema validator
- compatibility validator
- trace integrity validator
- replay validator
- leaderboard eligibility validator

## 3. Schema validation

Schema validation checks whether an artifact conforms to its declared JSON schema.

Schema validation MUST run on:

- scenario files
- world-state examples
- primitive catalogs
- tool manifests
- traces
- score reports
- submissions

## 4. Compatibility validation

Compatibility validation checks version and contract compatibility.

Examples:

- scenario version compatible with benchmark version
- trace generated by compatible scaffold version
- score report references valid evaluator versions

## 5. Trace integrity validation

Trace integrity validation SHOULD verify:

- required fields are present
- turn ordering is valid
- simulation time is monotonic
- tool responses align with tool requests
- final evaluation matches emitted score report
- state snapshot references are consistent

## 6. Replay validation

Replay validation SHOULD attempt deterministic reconstruction of a run from:

- scenario id and version
- benchmark version
- run seed
- action trace
- runner version

Replay validation SHOULD compare:

- final state hash
- final score
- event sequence

## 7. Leaderboard eligibility validation

A run or submission is leaderboard-eligible only if:

- all required artifacts validate
- no critical runtime failure occurred
- no evaluator failure invalidates score trust
- contamination declaration is present
- repeat-count policy is satisfied

## 8. Validation output contract

Every validator SHOULD emit:

- `validator_id`
- `validator_version`
- `artifact_type`
- `status`
- `errors`
- `warnings`

Status values:

- `ok`
- `warning`
- `failed`

## 9. Severity model

Validation findings SHOULD use severity classes:

- `info`
- `warning`
- `error`
- `critical`

Critical validation failures SHOULD make a run non-leaderboard-grade.

## 10. Recommended validator pipeline

Recommended order:

1. schema validation
2. compatibility validation
3. trace integrity validation
4. replay validation
5. leaderboard eligibility validation

## 11. Audit logging

Validator outputs SHOULD be stored alongside run artifacts so reviewers can see:

- what passed
- what failed
- why a run was rejected or accepted
